# RFC2136 (BIND, Technitium)

```env
RELAYD_PROVIDER_BIND_TYPE=rfc2136
RELAYD_PROVIDER_BIND_SCOPE=local
RELAYD_PROVIDER_BIND_URL=192.168.1.5:53
RELAYD_PROVIDER_BIND_KEY_NAME=relayd.invalid.
RELAYD_PROVIDER_BIND_KEY=base64-encoded-key
RELAYD_PROVIDER_BIND_KEY_ALGORITHM=hmac-sha256.
RELAYD_PROVIDER_BIND_ZONES=example.com
```

- **Scope**: `local`
- **Requires**: TSIG key configuration on your DNS server.
- **Algorithm**: Defaults to `hmac-sha256.` (Ensure the trailing dot is included if overriding).

### Technitium DNS

Technitium fully supports RFC2136, but it requires specific security policies to allow `relayd` to both read (Zone Transfer) and write (Dynamic Update) to the zone.

#### Technitium Server Configuration Guide

To secure your DNS while allowing `relayd` to function, you must configure a TSIG key and apply it to both the Zone Transfer and Dynamic Update settings.

**1. Create a TSIG Key**

- Go to **Settings > TSIG** and create a new key.
- **Name:** `relayd.invalid.` (Using `.invalid.` is a best practice to prevent public DNS resolution collisions. **Note the trailing dot!**)
- **Algorithm:** `hmac-sha256`
- Generate the key and copy the Base64 secret to your `.env` file.

**2. Allow Zone Transfers (Required for reading the zone)**
`libdns` requires a zone transfer (AXFR) to read the current state of your records before updating them.

- Go to your Zone -> **Options** -> **Zone Transfer**.
- Set the radio button to **Allow**.
- Under **TSIG Key Names**, type your exact key name (e.g., `relayd.invalid.`) and hit Enter to add it to the list.
  > **Security Note:** Setting the radio button to "Allow" combined with a key in the TSIG list does **not** allow everyone. It restricts transfers _exclusively_ to requests signed by that specific TSIG key.

**3. Allow Dynamic Updates (Required for writing records)**

- Go to your Zone -> **Options** -> **Dynamic Update**.
- Set the radio button to **Allow**.
- Under **Security Policy**, add a new rule:
  - **TSIG Key Name:** `relayd.invalid.`
  - **Domain Name:** `*.example.com` (You **must** use a wildcard if `relayd` needs to update subdomains like `app.example.com`).
  - **Allowed Record Types:** `A, AAAA, TXT` or just `ANY`
- If `relayd` also needs to update the apex domain (`example.com`), add a second identical rule with the **Domain Name** set to exactly `example.com`.
